# 角色状态管理操作指南

> 百万字长篇的核心挑战：角色多、时间跨度大、状态变化复杂。没有系统化的状态管理，到了第200章就会忘记某个角色现在在哪、受没受伤、和谁的关系发展到了什么程度。本指南定义角色状态的更新时机、检查清单和操作方法。

---

## 一、每次写完一章后的必做操作

写完一章 → 用户确认 → **必须执行以下检查**，更新 `03-角色档案/角色/{角色名}/角色信息.md` 和 `03-角色档案/角色关系.md`：

### 检查清单

| # | 检查项 | 操作 |
|---|--------|------|
| 1 | 本章出场角色有哪些？ | 更新对应 `角色信息.md` 的 `lastAppearedChapter` |
| 2 | 有角色状态发生变化吗？（受伤/死亡/突破/黑化/失踪/回归） | 更新对应 `角色信息.md` 的 `status` + 追加 `stateHistory` |
| 3 | 有角色移动到新地点了吗？ | 更新对应 `角色信息.md` 的 `currentLocation` |
| 4 | 有角色之间的关系发生变化了吗？ | 更新 `03-角色档案/角色关系.md` 中对应关系详情（追加关系历史、调整张力值） |
| 5 | 有角色获得了新能力吗？ | 追加 `abilities` 到该角色的 `角色信息.md` frontmatter |
| 6 | 有角色经历了重大情感转折吗？ | 更新 `角色信息.md` frontmatter 的 `personality` 相关字段（如 fear/desire 变化） |

## 二、状态字段详解

### `status` — 角色生死/活跃状态

| 值 | 含义 | 使用时机 |
|----|------|----------|
| `活跃` | 正常状态，参与剧情 | 默认状态 |
| `重伤` | 受伤失去行动能力 | 战斗中受到重创后 |
| `闭关` | 修炼/学习/自我提升中 | 暂时退出剧情 |
| `囚禁` | 被关押/控制 | 被捕/被困 |
| `失踪` | 下落不明 | 角色消失，其他角色在寻找 |
| `黑化` | 转向黑暗面 | 重大事件后性格剧变 |
| `洗白` | 从反派转向正派 | 重大事件后立场转变 |
| `死亡` | 确认死亡 | 该角色后续不再出现 |
| `离开` | 暂时/永久离开故事舞台 | 角色退场但不死亡 |

**规则**：
- 同一个角色可以多次在 `活跃` ↔ `重伤`/`闭关`/`囚禁` 之间切换
- `死亡` 是最终状态，一旦标记死亡不能复活（除非有复活设定，此时应有伏笔支撑）
- `黑化`/`洗白` 需要在前几章有铺垫，标记后追加到 `stateHistory`

### 角色当前状态维护（三个同步更新点）

`currentState` 字段已废弃。角色当前状态现在维护在三个位置，每次更新需保持同步：

| 位置 | 维护内容 | 示例 |
|------|---------|------|
| `角色信息.md` body → "角色动机演进" | 按阶段记录目标变化，每5-10章追加一条 | "第11-15章：发现师父遗书，转向寻找'最信任的敌人'" |
| `角色信息.md` body → "性格层次" | 如有性格层面变化，更新对应层 | "底层：从'压抑愤怒'变为'开始怀疑自己的判断'" |
| `08-写作动态.md` → "活跃角色"表格 | 每章更新：状态、当前位置、当前目标 | "张三 \| 活跃 \| 藏经阁 \| 解读遗书" |

每章写完后检查：
- **身体状态**：是否受伤、修为等级、健康程度 → 反映在 `status` 字段 + 写作动态"状态"列
- **情绪状态**：当前主导情绪 → 反映在写作动态"当前目标"列（如"愤怒中寻找真相"）
- **当前目标**：角色此刻最想完成的事 → 直接写入写作动态"当前目标"列

### `currentLocation` — 当前位置

必须具体到可操作的场景级别，不能模糊。

| ❌ 太模糊 | ✅ 正确写法 |
|----------|-----------|
| "在赶路" | "前往青云宗的官道上，距离宗门约三日路程" |
| "在城里" | "长安城西市·悦来客栈天字三号房" |
| "在外面" | "苍茫山脉外围·黑风峡谷入口处" |

**更新时机**：只要角色从一个场景移动到另一个场景，本章就必须更新。

### `lastAppearedChapter` — 最后出场章节

**作用**：
- 检测"被遗忘的角色"——如果某个主要角色超过30章没出现，需要评估是否该安排出场或调整计划
- 帮助判断关系是否停滞——超过20章没有关系变化的角色对，需要检查

### `stateHistory` — 状态变更日志

每次 `status` 或 `currentLocation` 发生变化时追加一条记录：

```json
{
  "chapter": 23,
  "status": "重伤",
  "location": "黑风峡谷·崖底暗河",
  "note": "与蝎尾蟒战斗后坠崖，左臂骨折，灵力耗尽"
}
```

## 三、关系网络管理

> 角色间的关系统一存储在 `03-角色档案/角色关系.md` 中，每个关系有独立的 `rel_N` 详情区块。

### 关系字段说明

| 字段 | 说明 |
|------|------|
| **类型** | 师徒/恋人/敌人/朋友/家人/盟友/暗恋/利用/前任/陌生人 |
| **动态** | 亲密信任/亦敌亦友/相爱相杀/一方暗恋一方不知/表面友好暗流涌动/不共戴天/利用与被利用/暧昧期/已确定关系 |
| **当前张力** | 1-10分，1=完全和谐，5=暗流涌动，10=一触即发 |
| **关系历史** | 按章节记录关键事件（时间线列表） |
| **最近重大变化** | "第X章 — 发生了什么变化" |
| **未来走向** | 计划的后续关系演变 |
| **关联伏笔** | 与该关系相关的伏笔ID |

### 关系更新时机

以下任一情况发生时，必须更新 `03-角色档案/角色关系.md` 中对应的关系详情：
1. 两个角色之间发生了**直接影响对方的事件**（救助/伤害/背叛/表白/决裂）
2. 两个角色**长时间分别后重逢**
3. 其中一方的**身份/立场发生变化**（如盟友其实是卧底）
4. 关系**积累了足够量变**（如连续5章的互动后，从"陌生"变为"初步信任"）

### 关系演化示例

在 `03-角色档案/角色关系.md` 中：

```markdown
### rel_002: 林渊 ↔ 沈清月 — 恋人

- **类型**: 恋人
- **动态**: 暧昧期
- **当前张力**: 6/10
- **关系历史**:
  - 第3章: 初见——她是追杀他的刺客组织成员
  - 第12章: 她违反组织命令放了他，两人之间出现裂缝般的信任
  - 第18章: 联手对敌时他为她挡了一剑。她问"为什么"
  - 第25章: 她脱离组织，两人在雨夜中同行，张力从杀意转为另一种东西
- **最近重大变化**: 第25章 — 她离开刺客组织，关系从敌人转为暧昧
- **未来走向**: 第45章左右确定关系，但第60章将面临身份揭露的考验
- **关联伏笔**: fs_007
```

## 四、角色状态批量更新流程

每章写完后执行以下操作步骤：

### Step 1：识别本章涉及的角色

从本章内容中提取：
- 出场的所有命名角色
- 虽然未出场但被提及状态变化的角色（"听说XXX已经突破了"）

### Step 2：逐角色检查变更

对每个涉及的角色，在 `03-角色档案/角色/{角色名}/角色信息.md` 中查找其档案，检查以下字段是否需要更新：

```
□ status          — 是否有状态变化？
□ currentLocation — 位置是否移动？
□ lastAppearedChapter — 更新为本章号
□ abilities       — 是否获得新能力/突破？
□ personality     — 性格/恐惧/欲望是否有变化？
□ stateHistory    — 如果有任何变化，追加一条记录
□ 正文"角色动机演进" — 如有阶段性目标变化，追加条目
□ 正文"性格层次"     — 如有性格层面变化，更新对应层
```

同时检查 `03-角色档案/角色关系.md`：
```
□ 本章涉及的角色对，关系是否有变化？
□ 如有变化，追加"关系历史"条目，调整"当前张力"
□ 更新"最近重大变化"
```

### Step 3：检查"被遗忘的角色"

每10章（第10、20、30...章）额外检查：
- 哪些主要角色（role=主角/反派/盟友/恋人）超过20章没有出场？
- 哪些关系超过20章没有在 `角色关系.md` 中追加新的"关系历史"条目？
- 在写作计划中标记：是否需要安排这些角色的回归。

### Step 4：保存

更新 `03-角色档案/角色索引.md` 的 `lastUpdated` 为当前时间。

## 五、角色档案更新示例

### 写第15章前 —— `03-角色档案/角色/林渊/角色信息.md` frontmatter：

```yaml
id: char_001
name: 林渊
status: 活跃
currentLocation: 青云宗外门·弟子宿舍甲区
lastAppearedChapter: 14

stateHistory:
  - { chapter: 0, status: 活跃, location: 初始位置, note: 故事开始前 }
```

### 第15章写完后（主角被偷袭受伤 + 被救走转移到山洞）

```yaml
id: char_001
name: 林渊
status: 重伤
currentLocation: 青云宗后山·隐秘山洞（被神秘人转移至此）
lastAppearedChapter: 15

personality:
  core: [坚韧, 隐忍, 重情]
  fatalFlaw: 过度信任身边人
  fear: 再次被至亲之人抛弃
  desire: 为家族洗刷冤屈

stateHistory:
  - { chapter: 0, status: 活跃, location: 初始位置, note: 故事开始前 }
  - { chapter: 15, status: 重伤, location: 青云宗后山·隐秘山洞, note: 遭黑煞掌偷袭，右臂经脉受损，被神秘人救走 }
```

正文区同时更新：
- "角色动机演进"追加第15章条目
- 如"性格层次"有变化，同步更新

## 六、常见错误与纠正

| ❌ 常见错误 | ✅ 正确做法 |
|------------|------------|
| 写完10章才想起来更新角色档案 | 每章写完后立即更新 |
| `currentLocation` 写成"在城里" | 具体到城市名+区域+建筑 |
| 只更新主角的状态 | 所有出场角色都检查 |
| 关系万年不变 | 每5-10章至少有一次关系推进 |
| 角色失踪50章也没标记 | 超过20章未出场→标记计划回归或调整状态 |
| 状态变化不记录日志 | 每次变更追加 `stateHistory` |

## 七、与Phase 3的协作

Phase 3 写作流程中：

**写前准备阶段**：
- 读取 `03-角色档案/角色索引.md` 定位出场角色
- 读取 `03-角色档案/角色/{角色名}/角色信息.md` 获取角色详情
- 读取 `03-角色档案/角色关系.md` 确认关系张力
- 确认每个出场角色的当前位置和状态，确保剧情衔接无缝

**写后更新阶段**（用户确认章节后）：
1. 保存txt文件
2. 更新 `00-剧情梗概.md`
3. **更新 `03-角色档案/角色/{角色名}/角色信息.md`**（按本指南的检查清单逐项更新）
4. **更新 `03-角色档案/角色关系.md`**（如有关系变化）
5. **更新 `03-角色档案/角色索引.md`**（同步状态变更）
6. **更新 `08-写作动态.md`**（刷新快照）
7. 更新 `04-伏笔清单.json`
8. 更新 `05-写作计划.json`
